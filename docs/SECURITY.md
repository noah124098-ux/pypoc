# Security Report — pypoc NSE Trading Agent

**Generated:** 2026-06-02  
**Scope:** Full codebase — `core/`, `backtest/`, `mcp_server/`, `dashboard.py`, `cli.py`

---

## Architecture Security Model

### DATA-ONLY constraint on Angel One credentials

Angel One SmartAPI credentials in this repo are exclusively for market data (ticks
and OHLC history). Order-placing SDK methods (`placeOrder`, `modifyOrder`,
`cancelOrder`, `placeOrderFullResponse`) are overwritten with a `RuntimeError`
stub immediately after every login, in three independent modules:

| Module | Guard location |
|---|---|
| `core/data/angelone_feed.py` | `_neutralize_order_methods()` called in `_connect()` |
| `core/data/angelone_history.py` | loop over `forbidden` set in `login()` |
| `core/data/angelone_portfolio.py` | `_neutralize_order_methods()` called in `__init__` |

The contract is enforced by `test_order_methods_are_neutralized_after_login` in
`tests/test_angelone_history.py`. Removing or bypassing the guard causes a test
failure that blocks any commit.

### No order execution possible via data-feed credentials

The `PaperBroker` (`core/broker/paper_broker.py`) is the only `IBroker`
implementation that is wired into the live orchestrator. It simulates fills
in memory; it never calls any broker API. A future live-broker module must use
separate credentials with its own DATA-ONLY audit before being enabled.

### Paper trading only until gate passes

`cli.py cmd_run` reads `data/backtest_gate.json` and refuses to start if the
gate is absent or older than 30 days. The gate requires Sharpe ≥ 1.2, MaxDD ≤ 15%,
win rate ≥ 45%, profit factor ≥ 1.5, ≥ 100 trades, and ≥ 3-year walk-forward.
This prevents premature live deployment.

### All orders pass through Guardrails.check() — no bypass path

Every signal, whether generated by a strategy or injected via the MCP command queue
(`place_paper_order`), passes through `Guardrails.check()` before being submitted to
the broker. The check enforces 14 hard rules including position limits, daily-loss
circuit (-3%), drawdown circuit (-10%), and black-swan halts.
`tests/test_guardrails.py` covers all 14 rules; `tests/test_integration.py` verifies
that the 5-position cap is enforced end-to-end.

## Audit Results (automated, 2026-06-02)

### Credential Handling

**Finding: PASS.** All credentials are read exclusively from environment variables
(`.env` via `python-dotenv`) and never appear in source files, logs, or YAML configs.
`core/config.py` uses `os.getenv(...)` for every secret field. `.gitignore` excludes
`.env`, `.env.*`, `*.pem`, `*.key`, `secrets/`, and `credentials.json`. A scan of all
Python source files for `logging.*` calls containing credential field names returned
zero matches — credential values are never logged.

The Streamlit dashboard stores notification credentials (Telegram token, SMTP
password) in `st.session_state` (in-process memory only); they are never written to
disk or logs.

**Finding: NOTE.** Two separate Angel One credential sets exist in `.env`:
`ANGEL_ONE_*` (data feed) and `ANGEL_ONE_LIVE_*` (future live broker). The live
credential set is defined but not yet wired to any active order path. This separation
is intentional and correct.

### Injection Vulnerabilities

**SQL — one low-severity finding in `dashboard.py`:**

Line 235 in `_get_signals()` constructs a query using an f-string:

```python
f"SELECT * FROM signals {where} ORDER BY id DESC LIMIT {limit}"
```

`where` is hardcoded to either `""` or `"WHERE accepted = 1"` (never derived from
user input), and `limit` is an integer constant (200). No user-supplied data enters
this query. Risk is theoretical — the function has no exploitable injection path
with the current callers — but the pattern should be avoided.

All other SQL in the codebase (`core/persistence/store.py`, `core/llm/eod_reviewer.py`,
`core/analytics/`) uses parameterized `?` placeholders consistently.

**Command injection — NONE.** A search for `subprocess`, `eval()`, and `exec()` in
`core/` returned no matches (only `evaluate()` method names, which are unrelated).

**YAML injection — NONE.** All YAML loading uses `yaml.safe_load` (verified in
`core/config.py` and `dashboard.py`). No `yaml.load()` without `Loader` is present.

### API Security

**Outbound HTTP — PASS.** All `requests.get()` calls in `core/data/` specify a
`timeout` parameter. No calls use `verify=False` or disable SSL certificate checking.
NSE and Upstox public endpoints are accessed over HTTPS only.

**MCP command queue — PASS.** The orchestrator enforces an explicit type allowlist
(`_ALLOWED_COMMAND_TYPES`) before processing any command from `data/commands.jsonl`:

```python
frozenset(["halt_agent", "resume_agent", "update_risk_param",
           "place_paper_order", "reload_config"])
```

Commands with unknown types are rejected and logged. String fields from commands
(e.g., halt `reason`) are truncated to 200 characters before use to prevent
memory or snapshot bloat.

**MCP server — read-only.** The MCP server exposed by `mcp_server/` provides 10
read-only tools. All mutations go through the file-based command queue, which is
then processed in the orchestrator's trusted process.

**Streamlit dashboard — no authentication.** The dashboard has no login gate. See
Known Risks below.

### Dependencies

**Pinned versions:** Most dependencies in `requirements.txt` are pinned to exact
versions (e.g., `pandas==2.2.3`, `pydantic==2.10.3`). `requests`, `python-dotenv`,
`streamlit`, `mcp`, and `pillow` use `>=` lower bounds.

**Notable packages and their security relevance:**

| Package | Version | Notes |
|---|---|---|
| `smartapi-python` | 1.5.5 | Angel One SDK; DATA-ONLY guards mitigate order-related risks |
| `anthropic` | 0.40.0 | Claude API client; API key in `.env` only |
| `pyyaml` | 6.0.2 | Only `safe_load` used; no arbitrary deserialisation |
| `streamlit` | ≥1.54.0 | No auth layer; mitigate with network-level controls |
| `python-telegram-bot` | 21.7 | Token in `.env` only |
| `mcp` | ≥1.23.0 | Exposes read-only tools; mutations gated by command queue |

No automated vulnerability scan (e.g., `pip-audit`, `safety`) is currently wired
into CI. Adding one to the test pipeline is recommended.

## Security Hardening Applied

- **Command queue type whitelist:** Unknown command types are rejected before any
  processing. Only five types are accepted.
- **Command queue field length caps:** The halt `reason` field is capped at 200
  characters (`orchestrator.py:486`).
- **SQL parameterized statements:** All queries in `core/` use `?` placeholders.
  One f-string query in `dashboard.py` has no user-input path (low risk, tracked).
- **YAML safe loading only:** `yaml.safe_load` used everywhere; `yaml.load` is
  absent from the codebase.
- **Credentials never logged:** Confirmed by source scan. `core/config.py` omits
  credential fields from `get_config_summary()`.
- **Credentials stored in `.env` only:** `.gitignore` excludes `.env` and all
  secret-file patterns.
- **Session state for dashboard credentials:** Telegram/SMTP secrets entered in the
  dashboard UI live only in `st.session_state` for the session lifetime.
- **DATA-ONLY guards with test enforcement:** Three independent guard points;
  one automated test; removing any guard breaks CI.
- **Backtest gate pre-flight:** Live trading is blocked until a fresh, passing
  walk-forward result is present.

## Known Risks

| Risk | Severity | Mitigation |
|---|---|---|
| DATA-ONLY guard depends on code correctness, not OS-level isolation | Medium | Three independent guard points + automated test; no OS sandbox available on Windows |
| Streamlit dashboard has no authentication | High | Deploy behind a VPN or nginx with HTTP basic auth; do not expose port 8501 to the public internet |
| SQLite database is unencrypted | Low | Contains trade data only, no credentials; acceptable for paper-trading use |
| Windows EC2 has no OS-level secret management (no KMS, no Vault) | Medium | Secrets in `.env` file; restrict file permissions to the Administrator account; rotate keys if EC2 is compromised |
| `data/commands.jsonl` is a plain file accessible to any local process | Low | Only meaningful if an attacker already has local code execution; acceptable for single-user deployment |
| `requirements.txt` uses `>=` bounds for some packages | Low | Pin to exact versions in a production deployment; add `pip-audit` to CI |
| No rate-limiting on MCP server | Low | Server is intended for local/LAN use only; not exposed publicly |
